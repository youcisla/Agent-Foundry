# Community Cohesion — Why some clusters look "loose"

Generated by the knowledge-graph audit on 2026-07-20.

## What the cohesion score means

The graph runs Leiden-style modularity optimization. **Cohesion** measures how strongly connected the nodes in a community are *to each other* (vs. how much they connect to other communities). A community with cohesion 0.6 is genuinely tight; one with 0.05 is a grab-bag.

A low cohesion score does **not** mean the cluster should be split. It usually means the cluster contains **cross-cutting infrastructure** that legitimately touches many parts of the system. Splitting it would *increase* the count of cross-community edges, not reduce it.

## How to use this list

- **Do not act on the low-cohesion communities** unless you can show the cross-cutting role is not actually needed.
- **For Communities 0 and 2 specifically**, the `Config` and `cli` modules are *the* integration surface. Splitting them would *fragment* the integration contract, not simplify it.
- **For Communities 1, 6, 7, 8, 10, 11**, the "communities" are artifacts of the AST extracting JSON schema values as orphan nodes. These will always show up. They can be suppressed by adding `enum_values` to the AST extractor's ignore list (out of scope for v0.2).
- **If you want to find real coupling smells**, look at the surprising connections list, not the cohesion scores.

## What low cohesion would actually look like (none of these are present)

A genuinely bloated module would show:

- Many unrelated symbols in one community
- God nodes with only one or two connections
- Surprising connections going *out* of the community to many other places

Agent Foundry does not show any of these. The current graph is healthy.
